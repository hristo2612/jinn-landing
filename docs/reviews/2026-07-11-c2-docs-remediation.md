# C2 documentation accuracy remediation — 2026-07-11

## Result

The stable documentation now targets the package a new install actually receives: `jinn-cli@0.25.0`. The authored tree contains 27 pages, every `since` value is at or below `0.25.0`, and the changelog and both machine routes name `0.25.0`.

The runtime contract gate installs the exact npm release in a disposable home, starts its daemon directly on an ephemeral high loopback port, executes 48 documented HTTP contracts, and removes the process and home in `finally`. It also executes six negative release-boundary probes to prove that removed `0.26.0` APIs remain absent from the stable release.

Final runtime result: **48/48 documented checks exact**, plus **6/6 unreleased-route probes returning the expected `404 {"error":"Not found"}`**.

## Stable machine contract

From `/agents.md` alone, an agent can discover the gateway through operator-provided `gateway.json` values, authenticate with the bearer token, pair a browser, discover engines and limits, create direct or employee sessions, create child sessions with `parentSessionId`, poll bounded message tails, send follow-ups, enumerate children, upload and attach managed files, inspect cron jobs, trigger an authorized job, and handle exact error/status boundaries. It also states what the stable gateway does not enforce: no Todo/Workflow/approval API, no atomic delegation/idempotency layer, and no built-in Jinn MCP server.

## Release-automation seam

`src/data/release.json` remains the single site release marker. Future release automation can update that file and append one changelog page; the runtime contract will fail until npm `latest`, frontmatter, machine routes, and the installed API agree.
